# Walkthrough: Map-Reduce with `Send()`

> **Notebook:** `map-reduce.ipynb`
> **Goal:** Use the `Send()` API to spawn a **dynamic** number of parallel branches at runtime — one per item in a list — and reduce all their outputs back into a single value.

---

## The Big Picture

Notebook 01 (parallelization) showed **static** fan-out: at compile time you draw edges from `a` to `b` and `c`, and from then on the graph always runs `b` and `c` in parallel.

That's not enough for problems shaped like *"the LLM just produced a list of N items — process every one of them in parallel"*. You don't know `N` until runtime. The number of parallel branches has to be *generated by a node*, not hard-coded into the topology.

LangGraph's solution is the **`Send()`** primitive. From a conditional edge you return a list of `Send(node_name, payload)` instructions, one per branch you want to spawn. LangGraph dispatches them in parallel, each running its own copy of the target node.

**Map-reduce in LangGraph is exactly this shape:**

```
   generate_topics ─┐
   (one node)       │  emits N subjects
                    ▼
    [Send → generate_joke(subject_1)] ─┐
    [Send → generate_joke(subject_2)] ─┤  ── all writes ─→  jokes (REDUCER REQUIRED)
    [Send → generate_joke(subject_3)] ─┤
    [Send → generate_joke(subject_N)] ─┘
                    │
                    ▼
                 best_joke
                  (reduce)
```

**Map** = one parallel branch per subject. **Reduce** = `best_joke` reads the merged `jokes` list and picks one.

> ### 📌 Reducers — the rule of this module
>
> *A reducer controls how a new value is merged into the existing value for a key. You need one when you either want accumulation, or when multiple nodes write the same key concurrently and you can't afford to lose any write.*
>
> Map-reduce is the **textbook concurrent-write case**. N parallel `generate_joke` branches all write `jokes` in the same super-step. Without `Annotated[list, operator.add]` on `jokes`, LangGraph hard-errors with `INVALID_CONCURRENT_GRAPH_UPDATE`.

---

## 1. The Toy Problem

The notebook builds a tiny "best-joke generator":

1. **Map** — given a topic, generate a list of sub-topics, then write one joke per sub-topic, **in parallel**.
2. **Reduce** — pick the funniest joke from the list.

We use Mistral for both LLM calls. Three prompts, all string templates:

```python
from langchain_mistralai import ChatMistralAI

subjects_prompt   = """Generate a list of 3 sub-topics that are all related to this overall topic: {topic}."""
joke_prompt       = """Generate a joke about {subject}"""
best_joke_prompt  = """Below are a bunch of jokes about {topic}. Select the best one!
Return the ID of the best one, starting 0 as the ID for the first joke. Jokes:

{jokes}"""

model = ChatMistralAI(model="mistral-medium", temperature=0)
```

`temperature=0` keeps the runs deterministic for testing.

---

## 2. State and Structured Outputs

Two Pydantic models for the LLM's structured output, plus a `TypedDict` for graph state:

```python
import operator
from typing import Annotated
from typing_extensions import TypedDict
from pydantic import BaseModel

class Subjects(BaseModel):
    subjects: list[str]

class BestJoke(BaseModel):
    id: int

class OverallState(TypedDict):
    topic: str
    subjects: list
    jokes: Annotated[list, operator.add]   # ← THE reducer
    best_selected_joke: str
```

**The four fields, audited:**

| Key | Writers | Reducer | Why |
|-----|---------|---------|-----|
| `topic` | the caller, once | none | Single write at `invoke()` time |
| `subjects` | `generate_topics` only | none | One writer per step, replace is fine |
| `jokes` | **N parallel `generate_joke` branches** | **`operator.add`** | Concurrent writes → reducer required |
| `best_selected_joke` | `best_joke` only | none | One writer per step |

`jokes: Annotated[list, operator.add]` is the entire reason this notebook works. Every parallel `generate_joke` returns `{"jokes": [its_joke]}`, and `operator.add` concatenates them all into a single list.

`Subjects` and `BestJoke` are **not** state — they're response shapes for `with_structured_output()`. They guarantee the LLM gives us a list of strings (not free-form prose) and an integer index (not "joke 2").

---

## 3. Generating Subjects (the input to map)

```python
def generate_topics(state: OverallState):
    prompt = subjects_prompt.format(topic=state["topic"])
    response = model.with_structured_output(Subjects).invoke(prompt)
    return {"subjects": response.subjects}
```

`with_structured_output(Subjects)` constrains the LLM to return a `Subjects` instance. `response.subjects` is then a regular `list[str]`. We write that list to state under `subjects`.

This node is the "input generator" of the map step. It produces the list that we'll fan out over.

---

## 4. The `Send()` API — Dynamic Fan-out

This is the core of the lesson:

```python
from langgraph.types import Send

def continue_to_jokes(state: OverallState):
    return [Send("generate_joke", {"subject": s}) for s in state["subjects"]]
```

**Read the type signature carefully.** `continue_to_jokes` is a *conditional edge function*, but instead of returning a string node-name (the usual contract), it returns a **list of `Send` objects**.

Each `Send(target_node, payload)` says: *"run `target_node` once, with this payload as its initial state."*

**Three things this lets you do that static `add_edge` can't:**

| Capability | Why it matters |
|------------|----------------|
| **Variable count** | `len(state["subjects"])` is not known at compile time. `Send()` figures it out at runtime. |
| **Per-branch payload** | Each branch sees a *different* `subject`. Static fan-out gives every branch the same state. |
| **Custom payload schema** | The payload doesn't have to match `OverallState`. `generate_joke` takes a tiny `JokeState`. |

---

## 5. The Map Step: `generate_joke`

```python
class JokeState(TypedDict):
    subject: str

class Joke(BaseModel):
    joke: str

def generate_joke(state: JokeState):
    prompt = joke_prompt.format(subject=state["subject"])
    response = model.with_structured_output(Joke).invoke(prompt)
    return {"jokes": [response.joke]}
```

**Two state schemas in one graph.** The `generate_joke` function is annotated with `JokeState`, which has only `subject` — not the full `OverallState`. That works because `Send()` is the one feeding it; `Send` can ship any payload it wants.

**Notice the return shape.** Every parallel call returns `{"jokes": [response.joke]}` — a **list of length 1**. That single-item list is the contribution this branch is adding to the merged `jokes` list. The `operator.add` reducer concatenates the N single-item lists into one final list of N jokes.

> **The pattern in one sentence:** *each parallel writer returns a list of its own contributions; the reducer concatenates the lists across writers.* Same shape we used for parallel web/wiki retrieval in notebook 01.

---

## 6. The Reduce Step: `best_joke`

```python
def best_joke(state: OverallState):
    jokes = "\n\n".join(state["jokes"])
    prompt = best_joke_prompt.format(topic=state["topic"], jokes=jokes)
    response = model.with_structured_output(BestJoke).invoke(prompt)
    return {"best_selected_joke": state["jokes"][response.id]}
```

By the time `best_joke` runs, **every parallel `generate_joke` has finished and the reducer has merged their outputs**. So `state["jokes"]` is the full list of N jokes.

The LLM picks an index (`response.id`), and we look it up in the list. This is the "reduce": collapse N items down to one.

---

## 7. Wiring It Up

```python
from langgraph.graph import END, StateGraph, START

graph = StateGraph(OverallState)

graph.add_node("generate_topics", generate_topics)
graph.add_node("generate_joke",   generate_joke)
graph.add_node("best_joke",       best_joke)

graph.add_edge(START, "generate_topics")
graph.add_conditional_edges(
    "generate_topics", continue_to_jokes, ["generate_joke"]
)
graph.add_edge("generate_joke", "best_joke")
graph.add_edge("best_joke", END)

app = graph.compile()
```

Two LangGraph-isms worth flagging:

- **`add_conditional_edges("generate_topics", continue_to_jokes, ["generate_joke"])`** — note the third argument. The list of *possible* targets is required because the conditional function returns `Send` objects whose target names LangGraph can't statically infer. You're telling the validator: *"this edge can only ever route to `generate_joke`"*.
- **`graph.add_edge("generate_joke", "best_joke")`** — even though `generate_joke` runs N times in parallel, you only declare *one* outgoing edge. LangGraph fans the N invocations back in to a single `best_joke` call, after the `jokes` reducer has merged everything.

## 8. Running It

```python
for s in app.stream({"topic": "animals"}):
    print(s)
```

Streamed output (one chunk per super-step):

```
{'generate_topics': {'subjects': ['mammals', 'reptiles', 'birds']}}
{'generate_joke':   {'jokes': ["Why don't mammals ever get lost? ..."]}}
{'generate_joke':   {'jokes': ["Why don't alligators like fast food? ..."]}}
{'generate_joke':   {'jokes': ["Why do birds fly south for the winter? ..."]}}
{'best_joke':       {'best_selected_joke': "Why don't alligators ..."}}
```

Three `generate_joke` chunks in a row — those are the three parallel branches. Their `jokes: [...]` outputs all merge into the single `jokes` list before `best_joke` sees it.

## 9. `Send()` vs Static Fan-out — When to Pick Which

| | Static fan-out (notebook 01) | `Send()` (this notebook) |
|---|---|---|
| Number of branches | Known at compile time | Decided at runtime |
| Edges | `add_edge("a", "b")`, `add_edge("a", "c")` | `add_conditional_edges("a", fn, ["target"])` |
| Branch input | Same state for every branch | Custom payload per `Send(target, payload)` |
| Branch state schema | Must be the parent's | Can be a private sub-schema |
| Typical use | Web + Wikipedia + RAG (fixed sources) | One LLM call per item, where N comes from data |

If `N` is data-driven, you almost certainly want `Send()`. If `N` is fixed and small, static fan-out is simpler.

---

## When the Reducer Bites You

Common gotchas around `jokes: Annotated[list, operator.add]`:

- **Forgot the reducer entirely** → first run hard-errors with `INVALID_CONCURRENT_GRAPH_UPDATE`. Add the reducer.
- **Returned a non-list** (e.g., `{"jokes": response.joke}`) → `operator.add` tries to concatenate a string with a list and explodes. Always return `{"jokes": [response.joke]}` from each map branch.
- **Reusing the same key name across notebooks** → the reducer attaches to the *key in this graph's schema*. Different graphs can have different reducers on the same-named key; they don't share.

---

## Key Takeaways

1. **`Send(target_node, payload)`** is how you spawn a runtime-determined number of parallel branches.
2. Return `Send` objects from a **conditional edge function** — and pass the third arg to `add_conditional_edges` so the validator knows the possible targets.
3. The branches LangGraph spawns from `Send()` can use **a different (smaller) state schema** than the parent graph — handy for keeping per-branch payloads clean.
4. Map-reduce *requires* a reducer on the key the map branches write to. `operator.add` is the standard list-concat reducer; each branch returns `{"key": [its_contribution]}`.
5. Fan-in is implicit: declare a single `add_edge("map_node", "reduce_node")` and the reduce node only runs once all map branches finish *and* the reducer has merged their writes.
6. **Use `Send()` when `N` is data-driven; use static fan-out when `N` is fixed.**
7. The `INVALID_CONCURRENT_GRAPH_UPDATE` error is your friend — it points at exactly the key whose merge rule is missing.
